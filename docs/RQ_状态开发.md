**兼容性，健壮性，鲁棒性，安全性，可扩展，可维护，高内聚低耦合。**

开发要求：尽可能采用英文命名，以兼容cmd等特殊情况。英文名称需要词义一致。能不简写不要简写。

### **鲁棒性**

- 避免：只注重功能实现，不注重实际应用性能。

- 需要：足够大量的测试（足够的tokens，足够的文档，足够的形态，足够的形式）

- 拒绝：应该，可能，大约，按理说。

例如：

- **解析word文档**：word2003，word2007版本，wps修改之后的版本，pdf转换为word的版本，md转换为pdf再转word的版本

- **模型Tokens测试**：100tokens的非流完成时间（可能成功），1000tokens非流完成时间（耗时>30s），2000tokens非流完成时间（实测就可能504）

### **可靠性**

- 避免：一次性成功
- 需要：自尝试、自恢复、自启动、自记录、自存储
- 拒绝：本机可行，换机器不可行，换环境，换场景，换设备不可行。

例如：

- 服务器：意外关机、断电、断网、504，404、非正常返回与重试
- 业务：本地异常关机、服务中断。任务中断状态保存与恢复执行。
- 不同的格式，不同的版本（doc,docx,wps,yaml,md）

- 文件不存在，目录不存在



### **常见性能开销**

1. nginx，没有连接池，速度很慢。

2. 数据库，没有连接池，速度很慢。

3. Python因为脚本解释GIL原因，实际上是单线程执行的，内核本质是没有多线程，一直都是单线程在运行（asyncio很重要，多进程也很重要，MCP网络通信等）。
4. Python密集型计算会降低整体程序的性能。pd和numpy做了专门优化，打破了GIL的线程锁。

# 可能的工具需求

## 可用工具（示例，可根据实际环境替换）

- `inspect_file(path)` — 返回文件类型、页数、是否含图片等元信息。
- `extract_text(path, page_range)` — 提取 PDF 文本内容。
- `ocr_page(path, page)` — 对图片型 PDF 页面进行 OCR，返回文字。
- `preview_excel(path, sheet)` — 预览目标 Excel 的表头和前几行数据。
- `search_excel_headers(path, keywords)` — 在 Excel 中搜索可能匹配的列名。
- `fill_excel(path, data, mapping)` — 根据映射关系将数据填入目标 Excel。
- `ask_user(question)` — 当无法自动判别时，向人类用户提出精确问题。
