# PageIndex 多文档检索优化方案：文档级预筛选 + 节点级 BFS

## 1. 优化背景

当前系统在 `GenCheckListState._selected_tree` 中直接对**森林（所有知识树的根节点）** 做 BFS 导航：

```
森林 BFS（节点级）：
  用户查询 → 对所有知识库的根节点做 BFS → 选出相关节点枝条 → 生成审查清单
```

**当知识库文档数量增加时（如 20+ 篇），此方案会出现以下问题：**
- **根节点爆炸**：每篇文档有 3~5 个根章节，50 篇文档就有 150~250 个根节点，LLM 一次性读不完（Token 超限）。
- **调用成本高**：每层循环都调 LLM，且大量节点最终被 `skip`，浪费算力。

---

## 2. 优化方案：双层 BFS

**核心思路**：把 BFS 拆成两层，先用文档级摘要快速筛选出相关文档，再对选中文档做节点级精准导航。

```
双层 BFS：
  用户查询 
    → 第一层（文档级）：阅读每篇文档的“描述”（30~50 字），选出 2~5 篇相关文档
    → 第二层（节点级）：只对选中文档的树节点做 BFS，按现有逻辑生成审查清单
```

---

## 3. 具体改动点

### 3.1 改动一：`BuildKnowledgeState` — 生成文档描述

在知识树构建完成后，增加一步：调用 LLM 生成文档级描述（30~50 字）。

**代码位置**：`BuildKnowledgeState._run_build` 末尾

```python
def _run_build(self, owner):
    # ... 原有构建逻辑：DFS 赋 ID/偏移、落盘 JSON 树 ...
    
    # ===== 新增：生成文档描述 =====
    try:
        # 读取刚落盘的 JSON 树，提取根节点摘要
        tree_path = self._output_dir / '知识树.json'
        tree_data = json.loads(tree_path.read_text(encoding='utf-8'))
        root_summaries = "\n".join([
            f"{n.get('number', '')} {n.get('title', '')}: {n.get('prefix_summary', '')}"
            for n in tree_data
        ])
        self.root_summaries = root_summaries
        
        actions = owner.execute_prompt('生成文档描述', self)
        description = actions[0]['params']['description']
        
        desc_path = self._output_dir / '文档描述.txt'
        desc_path.write_text(description, encoding='utf-8')
        GlobalFunction.log('BuildKnowledgeState', f'已生成文档描述：{description[:50]}...')
    except Exception as e:
        GlobalFunction.log('BuildKnowledgeState', f'生成文档描述失败：{e}')
```

**新增提示词模板**（`生成文档描述`）：

```text
你是一个文档摘要专家。请根据以下文档的根章节摘要，生成一句 30~50 字的文档概述，描述该文档的整体覆盖范围和适用范围。

根章节摘要：
{{root_summaries}}

要求：
- 突出文档的适用范围（如适用于什么类型的工程）
- 说明文档主要涵盖的内容方向
- 语言精炼，一句话概括

输出 JSON 格式：{"description": "..."}
```

### 3.2 改动二：`GenCheckListState` — 增加文档预筛选

在 `_run_generate` 的最前面，先加载所有文档的描述，让 LLM 选出相关文档，再加载选中的树做 BFS。

```python
def _run_generate(self, owner):
    try:
        # ===== 第一阶段：文档预筛选 =====
        doc_descriptions = []
        for name in self.knowledge_names:
            desc_path = Path(owner.workspace_path) / "knowledge" / name / "文档描述.txt"
            if desc_path.exists():
                desc = desc_path.read_text(encoding='utf-8').strip()
            else:
                # 兜底：用文档标题 + 一级目录拼接
                tree_path = Path(owner.workspace_path) / "knowledge" / name / "知识树.json"
                if tree_path.exists():
                    trees = json.loads(tree_path.read_text(encoding='utf-8'))
                    titles = [n.get('title', '') for n in trees[:3]]
                    desc = f"{name}：{'、'.join(titles)}"
                else:
                    desc = name
            doc_descriptions.append({
                "knowledge_name": name,
                "description": desc
            })
        
        # 如果文档数量 <= 3 篇，跳过预筛选，直接加载全部
        if len(doc_descriptions) <= 3:
            selected_names = self.knowledge_names
        else:
            self.doc_descriptions = json.dumps(doc_descriptions, ensure_ascii=False, indent=2)
            actions = owner.execute_prompt('文档预筛选', self)
            selected_names = actions[0]['params']['selected_knowledge_names']
            
            # 兜底：如果 LLM 返回空或格式不对，使用全部文档
            if not selected_names or not isinstance(selected_names, list):
                GlobalFunction.log('GenCheckListState', '文档预筛选返回空，回退到全部文档')
                selected_names = self.knowledge_names
        
        # ===== 第二阶段：加载选中的知识树，执行原有的森林 BFS =====
        all_trees = []
        for name in selected_names:
            tree_path = Path(owner.workspace_path) / "knowledge" / name / "知识树.json"
            if tree_path.exists():
                all_trees.extend(json.loads(tree_path.read_text(encoding="utf-8")))
        
        if not all_trees:
            GlobalFunction.log('GenCheckListState', '没有加载到任何知识树，流程终止')
            self._done = True
            return
        
        # 后续逻辑完全不变
        selected = self._selected_tree(owner=owner, trees=all_trees)
        self.knowledge_tree = selected
        actions = owner.execute_prompt("生成审查清单", self)
        checklist = actions[0]['params']['checklist']
        
        output_path = self._output_dir / '审查清单.json'
        output_path.write_text(
            json.dumps(checklist, ensure_ascii=False, indent=2),
            encoding='utf-8'
        )
    except Exception as e:
        GlobalFunction.log('生成审查清单错误', f'错误为{e}')
    finally:
        self._done = True
```

**新增提示词模板**（`文档预筛选`）：

```text
你是一个合规审查专家。用户提交的方案类型是：{{plan_type}}

现有以下知识库文档，每份覆盖不同的法规或标准范围：
{{doc_descriptions}}

请选出与 "{{plan_type}}" 最相关的 2~4 篇文档，只输出知识库名称列表。

判断原则：
1. 文档适用范围与方案类型直接匹配 → 必选
2. 文档涵盖通用性规定（如总则、基本规定）→ 可考虑入选
3. 文档适用范围与方案类型明显不同（如桥梁规范 vs 隧道方案）→ 不选
4. 宁可多选，不可遗漏

输出 JSON 格式：{"selected_knowledge_names": ["文档A", "文档B"]}
```

## 4. 优化前后对比

| 维度 | 优化前 | 优化后 |
| :--- | :--- | :--- |
| **筛选粒度** | 直接对节点做 BFS | 先选文档，再对选中文档的节点做 BFS |
| **LLM 调用次数** | N 次（每层循环） | +1 次（文档预筛选），但后续 BFS 规模大幅缩小 |
| **根节点输入量** | 所有文档根节点之和（如 50 篇 × 3 根 = 150 节点） | 仅选中文档根节点（如 3 篇 × 3 根 = 9 节点） |
| **Token 消耗** | 随文档数量线性增长 | 文档级预筛选仅消耗 50 字/篇 × 文档数，节点级 BFS 规模显著缩小 |
| **适用规模** | 文档数量 < 20 篇 | 文档数量可扩展至 100+ 篇 |
| **代码改动量** | — | 极小：`_selected_tree` 逻辑完全不动，只在 `_run_generate` 前面加一层 |

## 5. 实施建议

### 5.1 当前阶段（文档 ≤ 5 篇）
- **不需要实施此优化**，现有 BFS 完全够用。
- 但可以预先为现有文档生成 `文档描述.txt`，以备将来扩展。

### 5.2 中期阶段（文档 6~20 篇）
- **建议启用“文档预筛选”**，但阈值设为 5 篇以上才触发（避免单篇文档时多一次无效调用）。
- 预筛选的 LLM 调用成本很低（50 篇描述约 2500 字符，约 800 token），可接受。

### 5.3 长期阶段（文档 20+ 篇）
- **文档预筛选成为必须**，否则根节点数会直接撑爆上下文。
- 可进一步引入“按元数据搜索”（如按文档类型、发布日期过滤）作为更粗粒度的前置筛选。

---

## 6. 扩展思考

此优化方案的本质是**在现有 BFS 框架上增加了一个新的“入口层”**。同样的思路可以继续扩展：

- **第三层（按元数据）**：如果文档有明确的元数据标签（如文档类型、适用地区、发布日期），可以在文档预筛选之前再加一层元数据过滤。
- **第四层（按语义）**：如果文档数量达到数百篇，可以在元数据之后加一层向量语义检索，选出 Top-20 篇后再进入文档预筛选。

**核心设计原则**：每层筛选的目标都是“快速淘汰无关内容，缩小下一层的处理范围”，让 LLM 始终在可控的上下文窗口内工作。
