好的，我们跳出代码细节，来梳理一下**融合了任务分类、动态规划、意图确认的 ReAct 智能体的整体流程架构**。

---

## 一、核心设计哲学

整个架构的基石是一个三层循环：

1. **元循环**：处理意图不清时的澄清与背景收集。
2. **主循环**：ReAct 的“规划—执行—反思”循环，但规划不再是一次性的，而是按需触发。
3. **子循环**：信息依赖驱动时，自动插入的信息获取子任务。

三层并非物理隔离，而是通过**计划栈**和**信息库**统一调度。

---

## 三、流程主干：一次完整任务的经历

### 阶段 0：意图确认 (Intent Gate)

**触发条件**：任务描述模糊、关键实体缺失、分类器置信度低、或存在未解决的澄清标记。

**该阶段的行为**：
- 生成一个**意图确认计划**，而不是马上执行正式任务。
- 该计划包含：① 自动查询本地知识/用户档案/近期文档 → ② 如仍不足，生成精准问题 → ③ 向用户提问 → ④ 整合信息，判断是否就绪。
- 若就绪，清除标记，进入正式阶段；若仍不足，循环但限制轮次。

**关键点**：意图确认不是阻塞式“一问一答”，而是**自主捞背景 + 必要提问**的混合步骤。

---

### 阶段 1：任务分类与初始规划

**触发条件**：通过意图门，且当前无计划。

**该阶段的行为**：
1. **任务分类器**（可调用 LLM 或规则）将任务映射到 `knowledge_search | form_filling | generic_task | exploration` 等类别。
2. **规划器**根据类别加载对应的**规划模板**，结合当前 `info_store`（已包含部分本地背景和用户补充），生成具体的步骤序列（Plan）。
3. 该 Plan 作为“主干计划”压入计划栈。

---

### 阶段 2：ReAct 执行循环

此时进入常规的“步骤→动作→观察→反思”迭代，但已嵌入依赖检查和动态调整。

#### 每一步的内部微循环：

```
当前步骤 (Step)
    │
    ├─[检查依赖]─ 缺失信息？ ──是──→ 生成“信息获取子计划”
    │                                 插入当前步骤之前，重新进入
    │
    └─[决定动作] (Decide)
          │
          ├─ 根据当前步骤的 provider 和历史观察，产生具体 action
          │
    └─[执行动作] (Execute) ──→ 获得 Observation
          │
    └─[反思观察] (Reflect)
          │
          ├─ 符合预期？ ──否──→ 修改假设，可能重规划
          ├─ 触发了新任务？（如报错提示需实名认证）──是──→ 新任务压栈，生成新计划
          └─ 符合预期，步骤完成，更新 info_store，移动步骤指针
```

#### 其中关键子结构：

- **信息依赖检查**：在决定动作前，由代码直接检查步骤的 `requires` 中的信息项在 `info_store` 中是否存在。若缺失，生成一个轻量的“获取信息子计划”，其步骤是固定的：① 查本地 → ② 搜网络 → ③ 问用户。该子计划被注入当前步骤前，自动执行。
- **Decide 模块**：将计划、当前步骤、信息和历史观察喂给 LLM，输出一个带参数字的动作。对于确定性高的步骤（如 `browser:navigate`），可以直接用模板映射，无需 LLM。
- **Reflect 模块**：分析观察结果，判断是成功、预期外错误、还是触发新任务。这决定了是移动指针、原地重试、修补计划还是压入新任务。

---

### 阶段 3：任务完成或转换

- 当主干计划所有步骤 `completed`，且没有旧任务压栈，协调器输出最终结果。
- 如果在执行中弹出过新任务（如“先实名认证”），这些新任务以**栈**的形式管理，当前任务完成后自动恢复到上一级任务。

---

## 四、计划栈与多任务管理

这里正式引入**计划栈（Plan Stack）**：

- 初始主干计划入栈。
- 遇到信息缺失，生成子计划，插入当前计划内部（不改变栈）。
- 遇到**完全独立的新任务**（如提交时触发“认证”任务），新任务生成独立的 Plan，`push` 到栈顶。
- 当前 Plan 执行完毕则 `pop`，回到原来的任务继续。

这样保证了“中断—恢复”的清晰语义，且每一步都知道自己属于哪个任务上下文。

---

## 五、信息库 (Info Store) 的全局角色

`info_store` 是跨越所有阶段的**共享记忆**：

- 在意图确认阶段，本地背景资料被注入，标记来源 `local`。
- 在信息获取子计划中，用户回答、网络搜索结果、本地查询结果都被存入，携带来源和可信度。
- 在执行阶段，每个步骤的 `requires` 都从这里查询。
- 反思阶段，如果新信息与旧信息冲突，会触发更新或重规划。

信息库本质上是一个**类型化的键值存储 + 来源追踪**，避免 LLM 直接管理信息带来的差错。

---

## 六、不同任务类型的规划差异（一览）

| 任务类型     | 初始计划特点                              | 处理差异                                                 |
| ------------ | ----------------------------------------- | -------------------------------------------------------- |
| **知识搜索** | 搜索→整合→回答                            | 步骤少，几乎无信息依赖，直接 decide 生成搜索词           |
| **信息分发** | 识别信息→匹配模板→触发实际任务            | 实为跳板，匹配后常压入新任务（如登录排错）               |
| **填报类**   | 确定网址→收集字段→补齐信息→导航→填写→提交 | 重度依赖信息依赖检查，会有多次“信息获取子计划”插入       |
| **通用任务** | 从任务需求库直接取 SOP                    | 执行中若失败，Reflect 可能触发重规划或转入探索模式       |
| **业务摸索** | 假设→试探→观察→更新模型→循环              | 计划本身是**元探索循环**，每个动作前生成假设，观察后验证 |

---

## 七、整体架构的闭环逻辑

```
意图模糊 → 主动收集 + 精准提问 → 就绪
     ↓
任务分类 → 按类型产出主干计划
     ↓
进入执行循环：
   每一步：检查信息 → 缺则补 → 决定动作 → 执行 → 反思
     ↓
   若触发新任务 → 压栈，新计划
     ↓
   若完成 → 出栈，继续或结束
```

整个过程没有一次性全规划好所有细节，而是**框架规划 + 按需细化 + 信息驱动**。这正是你一直期望的：既有规划的前瞻性，又有 ReAct 的灵活性，还融入了不打断用户的意图确认机制。

如果需要，下一步可以讨论组件之间的接口定义，或者针对某个特定任务类型（比如“业务系统摸索”）的具体计划模板。



# 细节步骤0

实现“意图确认与丰富”的核心，是把模糊的用户输入，通过**本地背景自动收集 + 精准提问 + 就绪判断**，转换成清晰的、可执行的任务上下文。整个过程由一个轻量的`IntentClarificationPlan`驱动，完全嵌入在现有 `reasoning_next_action` 循环中。

下面给出可落地的**实现方案、提示词设计、以及代码闭环**。

---

## 一、Intent Gate 的判断逻辑（何时进入阶段0）

在 `reasoning_next_action` 入口处，用一个轻量函数判断是否需要澄清，避免过度进入该阶段。

```python
def is_intent_unclear(task_description: str, info_store: InfoStore) -> bool:
    # 条件1：info_store中有之前未解决的澄清标记
    if info_store.get("pending_clarification"):
        return True

    # 条件2：通过LLM快速判定模糊度（带置信度与缺失实体）
    clarity_check = llm_check_clarity(task_description, info_store)
    # clarity_check: {"confidence": 0.6, "missing_entities": ["tax_type","period"], "is_ambiguous": true}

    return clarity_check["is_ambiguous"] or clarity_check["confidence"] < 0.7
```

`llm_check_clarity` 是一个专用提示词（如下），它**不负责规划，只负责判断**当前任务描述结合已知信息是否能撑起后续操作。

### 提示词：`intent_clarity_check`

```
你是一个任务清晰度评估器。根据用户的任务描述和当前已有的背景信息，判断任务意图是否足够清晰，能否直接开始分类与规划。

已知信息：
{{info_store_summary}}

用户任务描述：
"{{task_description}}"

请返回JSON：
{
  "is_ambiguous": true/false,
  "confidence": 0.0-1.0,          // 对任务意图的确定程度
  "missing_entities": ["field1"],  // 如果模糊，缺失哪些关键信息
  "reason": "简要说明"
}

判定标准：
- 如果任务描述缺少必要的对象（什么税种、哪个系统、哪个文件等），则is_ambiguous=true。
- 如果已知信息可以补全这些对象，则confidence可较高，可能is_ambiguous=false。
```

这样，常规清晰任务不会进入阶段0，只有确实模糊的才会触发。

## 二、意图确认计划的构建 (`build_intent_clarification_plan`)

一旦判定需要澄清，就构建一个固定结构的计划，包含4个步骤：

| 步骤 | 意图                     | 执行者                                | 说明                                                  |
| ---- | ------------------------ | ------------------------------------- | ----------------------------------------------------- |
| 1    | 主动收集本地背景信息     | `local_context_retrieval`（工具）     | 不调用LLM，直接查询用户配置、近期文档、历史对话摘要等 |
| 2    | 生成精准澄清问题（如需） | `llm:generate_clarification_question` | 基于步骤1的结果和任务描述，决定是否提问、问什么       |
| 3    | 向用户提问               | `user:ask`                            | 若步骤2决定需提问，输出问题并等待回复                 |
| 4    | 整合信息并判断就绪       | `llm:decide_readiness`                | 综合所有信息，判断是否可以退出意图确认进入正式任务    |

**关键：** 步骤2可能会输出 `{"need_question": false}`，此时步骤3会被跳过。

### 代码实现

```python
def build_intent_clarification_plan(task_description: str, info_store: InfoStore) -> Plan:
    plan = Plan(plan_type="intent_clarification")
    
    # Step 1: 收集本地背景
    plan.add_step(Step(
        step_id="cl_1",
        intent="主动收集本地背景信息",
        requires=[],
        provider="tool:local_context_retrieval",
        params={"task_hint": task_description}
    ))
    # Step 2: 生成澄清问题 (LLM)
    plan.add_step(Step(
        step_id="cl_2",
        intent="基于本地背景决定是否需要向用户提问",
        requires=["cl_1.result"],
        provider="llm:generate_clarification_question",
        params={"task": task_description}
    ))
    # Step 3: 向用户提问 (条件执行)
    plan.add_step(Step(
        step_id="cl_3",
        intent="向用户提出澄清问题",
        requires=["cl_2.question_text"],  # 如果cl_2输出need_question=false，这个依赖会缺失，由此触发跳过逻辑
        provider="user:ask",
        params={"message": "{{cl_2.question_text}}"}
    ))
    # Step 4: 判断就绪
    plan.add_step(Step(
        step_id="cl_4",
        intent="整合信息判断是否足够执行任务",
        requires=["cl_1.result", "cl_3.user_response"],  # cl_3可能未执行，此时user_response为空
        provider="llm:decide_readiness",
        params={"task": task_description}
    ))
    return plan
```

步骤3的跳过机制：当步骤2决定不需要提问时，不会产出 `question_text`，步骤3的 `requires` 无法满足，依赖检查代码会直接将它标记为 `skipped` 并自动进入步骤4，步骤4也能看到 `cl_3` 没有用户回复，处理见后。

## 三、各步骤的提示词与工具设计

### Step 1: 本地背景收集（非 LLM，工具实现）

`local_context_retrieval` 工具不需要 LLM，直接从几个数据源拉取并聚合：

```python
def retrieve_local_context(task_hint: str) -> dict:
    context = {}
    # 用户档案：角色、部门、常用税号等
    context["user_profile"] = get_user_profile(current_user_id)
    # 组织默认配置：当前会计期间、默认申报网址等
    context["org_config"] = get_org_config()
    # 最近文档标题和摘要（向量相似搜索task_hint）
    context["recent_docs"] = search_recent_docs(task_hint, k=3)
    # 对话历史摘要
    context["recent_dialog_summary"] = summarize_last_n_dialogs(3)
    # 从info_store中取已有的关键实体
    context["previously_known_entities"] = info_store.get_known_entities()
    return context
```

结果存入 `info_store`，例如 `info_store.update("local_context", context)`，供后续 LLM 步骤使用。

---

### Step 2: 生成澄清问题 (`llm:generate_clarification_question`)

提示词设计要点：**不盲目提问，优先判断本地信息是否已足够**。

```
你是一个智能助手的意图澄清模块。你需要判断在当前背景下，是否还需要向用户提出问题以明确任务。如果必须提问，生成一个精准、具体的问题（可附带选项）；如果不需要，直接返回无需提问。

【任务描述】
{{task_description}}

【本地已获取的背景信息】
{{local_context}}

【已知信息库】
{{info_store_summary}}

请返回JSON：
{
  "need_question": true/false,
  "reasoning": "为什么需要/不需要提问",
  "question_text": "如果需要提问，生成的自然语言问题（可包含2-3个具体选项）",
  "options": ["选项A","选项B"]  // 可选，适合对话式界面
}

注意：
- 如果本地信息已经能明确锁定具体任务（如已有默认税号、对应期间和任务常用操作），则不需要提问。
- 如果需要提问，问题应当聚焦于仍然不确定的1-2个关键点，避免泛泛而问。
```

如果 `need_question: false`，后续步骤3会被跳过，信息不会丢失。

---

### Step 3: 向用户提问 (`user:ask`)

这是一个阻塞式的交互动作，由前端或对话系统实现。它会返回用户的回复字符串，存入 `info_store` 并关联到 `cl_3.user_response`。

---

### Step 4: 判断就绪 (`llm:decide_readiness`)

提示词：

```
你是任务就绪判断器。结合最初的任务描述、本地收集的背景信息、以及用户对澄清问题的回复（如果有），判断当前是否已经具备进入正式任务执行的条件。

【任务描述】
{{task_description}}

【本地背景信息】
{{local_context}}

【用户对澄清问题的回复】（可能为空）
{{user_response}}

请返回JSON：
{
  "is_ready": true/false,
  "confidence": 0.0-1.0,
  "resolved_task_description": "更完整清晰的任务描述（若就绪，用于后续分类规划）",
  "missing_aspects": ["..."]  // 若未就绪，仍然缺失的关键信息
}
```

如果 `is_ready: true`，则可以：
- 将 `resolved_task_description` 存入 `info_store` 或更新 `owner.task_description`。
- 清除 `pending_clarification` 标记。
- 本轮 `reasoning_next_action` 将计划清空，下一轮就会进入正式分类与规划。

如果 `is_ready: false`，则：
- 设置 `pending_clarification` 标记为 True（或者增加澄清轮次计数）。
- 如果未超过最大轮次（如2次），可以再次进入意图确认流程（可通过重新问一个更具体的问题）。
- 如果超过限制，则以较低置信度强行进入后续流程，如 `owner.task = "按最佳猜测执行"`，并通知用户。

## 四、代码闭环：嵌入 `reasoning_next_action`

```python
class Agent:
    MAX_CLARIFICATION_ROUNDS = 2

    def reasoning_next_action(self, owner):
        task = owner.task_description
        info_store = owner.info_store
        plan = owner.current_plan

        # --- Intent Gate ---
        if self.is_intent_unclear(task, info_store):
            # 检查是否超过澄清轮次
            if info_store.get("clarification_rounds", 0) >= self.MAX_CLARIFICATION_ROUNDS:
                # 强行继续，给出一个默认任务描述
                forced_task = self.force_resolve_task(task, info_store)
                owner.task_description = forced_task
                info_store.delete("pending_clarification")
                # 继续到正常分类规划
                return self._start_normal_task(owner)
            else:
                # 进入意图确认
                info_store.increment("clarification_rounds")
                plan = self.build_intent_clarification_plan(task, info_store)
                owner.current_plan = plan
                return self._decide_next_action(plan, info_store, owner.history, "intent_clarification")

        # --- 正式任务逻辑（无计划 → 分类规划；有计划 → 依赖检查、决定动作等） ---
        # ...
```

当意图确认计划执行完毕（步骤4返回 `is_ready: true`）后，在 `reflect` 阶段会识别计划完成，并清空 `current_plan`。此时下一轮 `reasoning_next_action` 重新进入 Intent Gate，但由于 `pending_clarification` 已清除，且任务描述变得清晰，`is_intent_unclear` 会返回 False，从而正常进入分类与规划。

## 五、效果演示

**用户输入**：“帮我报税。”  
**已知**：无额外信息。

1. `is_intent_unclear` 判定模糊（缺少税种、期间）→ 进入阶段0。
2. Step1：本地检索发现用户配置中默认税号 `91440101MA...`，最近修改过“2025-03 增值税申报底稿.xlsx”。
3. Step2：LLM 看到这些背景，生成问题：“根据您的记录，似乎是申报2025年3月的增值税，使用税号尾号1234。是否正确？如果收入数据已在桌面表格里，我可以直接提取。”
4. Step3：用户回复：“对，从表格取数据。”
5. Step4：LLM 整合后输出 `is_ready: true`，`resolved_task_description: "填报2025年3月增值税，从用户桌面表格提取数据，税号91440101MA..."`。
6. 意图确认计划完成，下一轮进入正常分类（`form_filling`）并规划填报步骤。
