# 铁路施工方案辅助审查系统 — 架构设计文档

## 阶段一：概念化

### 1.1 核心定位

**领域任务专家** — 聚焦铁路施工方案合规审查，追求在该场景做到极致可靠，而非面面俱到的通用助手。

### 1.2 定位金字塔

```
     ┌─────────────────────────┐
     │  铁路施工方案合规性审查    │  ← 核心场景
     ├─────────────────────────┤
     │  全流程辅助 · 人工确认节点 │  ← 工作模式
     ├─────────────────────────┤
     │  施工组织设计 / 专项方案   │  ← 审查对象
     ├─────────────────────────┤
     │  铁路标准·准则·条规       │  ← 知识底座
     │  (PDF/Word 文档形态)      │
     └─────────────────────────┘
```

### 1.3 全流程链路

方案上传 → 自动初审（标记不合规项） → 人工确认（修正/补充） → 生成报告（条款引用+建议） → 归档

### 1.4 关键约束

1. **辅助而非替代** — 每个环节保留人工确认，最终责任在审查员
2. **标准可溯源** — 每条审查意见必须引用具体标准条文出处
3. **知识库从文档构建** — 铁路标准以 PDF/Word 形态输入，需解析+结构化

## 阶段二：需求分析

### 2.1 用户场景

审查员收到一份铁路施工方案（docx/pdf），需要对照铁路行业标准逐条检查合规性。当前完全依赖人工翻阅标准文档，效率低且容易遗漏。

### 2.2 核心工作流

1. 审查员上传施工方案文档
2. 系统根据方案类型自动生成审查清单
3. 逐项检查：检索相关标准条文 → 比对方案内容 → 判定符合/不符合/存疑
4. 审查员复核不符合项和存疑项，做出最终确认
5. 系统生成结构化审查报告，归档备查

### 2.3 与非目标的边界

- **做**：基于已有标准的合规性检查
- **不做**：标准的自动更新/维护（由人工触发）
- **做**：标记违规项并引用条文
- **不做**：给出法律建议或最终裁决（那是审查员的职责）
- **做**：支持多种标准文档格式解析
- **不做**：支持所有格式（初期聚焦 docx/pdf）

## 阶段三：领域建模

### 3.1 核心领域实体

| 实体 | 定义 | 关键属性 |
|------|------|---------|
| **标准知识库** | 从铁路标准 PDF/Word 解析、结构化而来的条文集合 | 标准体系、条文编号、条文原文、类别（强条/应/宜/参考）、适用场景 |
| **审查清单** | 从知识库派生，驱动逐项检查的"体检表" | 检查项描述、关联条文编号、条文原文引用、强制度、适用章节 |
| **施工方案** | 被审查的工程文档 | 方案类型、章节结构、内容文本块 |
| **审查意见** | 每条检查项的判定结果 | 判定（符合/不符合/存疑/不适用）、违规条文引用、方案原文引用、判定理由、人工确认记录 |

### 3.2 实体关系

```
铁路标准文档 ─解析结构化→ 标准知识库
标准知识库   ─派生→       审查清单
审查清单 + 施工方案 ─逐项检查→ 审查意见
审查意见    ─汇总→       审查报告
审查意见    ─存疑项→     人工确认 ─反馈→ 审查意见
```

## 阶段四：关键需求

### R1 标准知识库构建

将铁路标准 PDF/Word 文档转化为可检索的结构化条文库。

- 文档解析（docx/pdf→结构化文本，复用已有 WordParseState）
- 条文层级提取（保留篇→章→节→条→款结构）
- PageIndex 语义树索引构建（用于 LLM 导航检索）
- 分类标注（强条/应/宜/参考）
- 版本管理（标准更新时重建索引）

### R2 审查清单管理

从知识库中筛选、组织出针对特定方案类型的审查清单，保证审查覆盖率。

- 按方案类型筛选适用条文（桥涵/路基/隧道/...）
- 检查项与条文关联（每条检查项可追溯到条文原文）
- 强制度标识（强条优先检查）
- 清单可编辑/可补充（人工可增删检查项）

### R3 逐项合规检查

LLM 依据审查清单逐条检查施工方案，判定合规性并引用具体条文。

- 方案解析（docx/pdf→结构化文本）
- PageIndex 树导航定位相关条文原文
- 清单项 ↔ 方案段落语义匹配
- 判定：符合 / 不符合 / 存疑 / 不适用
- 违规项标注具体条文出处 + 方案相关段落引用

### R4 人工复核闭环

存疑项和不符合项提交人工确认，复核结果反馈回系统。

- 存疑/不符合项列表（附 AI 判定+依据）
- 人工标注：确认 / 驳回 / 补充
- 驳回时修正审查意见
- 记录复核历史（谁、何时、做了什么判断）

### R5 审查报告生成

汇总所有检查项判定结果，生成结构化审查报告并归档。

- 审查概览（总项数、通过项、不符合项、存疑项）
- 不符合项明细表（每项附条文引用）
- 报告格式：结构化数据 + 可导出
- 按项目/日期归档

### 非功能需求

| 需求 | 说明 |
|------|------|
| 可追溯 | 每条审查意见 → 对应标准条文编号，可供人工验证 |
| 覆盖完整 | 审查清单保证所有适用条款都被检查，不遗漏 |
| 知识可更新 | 标准更新后重新解析入库，审查清单同步刷新 |
| 人工兜底 | 不确定的地方不硬判，明确标记"存疑"等人工 |

## 阶段五：概念架构设计

### 5.1 五层架构

```
┌─────────────────────────────────────────────┐
│              离线阶段（标准更新时）             │
├─────────────────────────────────────────────┤
│  第一层：知识层                              │
│  标准PDF/Word → 文档解析 → 条文提取           │
│  → PageIndex 语义树（一次性构建）              │
├─────────────────────────────────────────────┤
│  第二层：清单层                              │
│  选择方案类型 → LLM 筛选条文 → 审查清单 JSON    │
├─────────────────────────────────────────────┤
│              在线阶段（每次审查）               │
├─────────────────────────────────────────────┤
│  第三层：执行层                              │
│  上传方案 → 文档解析 → 逐项检查循环             │
│  （PageIndex 导航 → LLM 判定 → 记录结果）      │
├─────────────────────────────────────────────┤
│  第四层：复核层                              │
│  存疑项+不符合项 → 人工标注 → 修正审查结果      │
├─────────────────────────────────────────────┤
│  第五层：报告层                              │
│  汇总判定 → 生成审查报告 → 归档               │
└─────────────────────────────────────────────┘
```

### 5.2 与现有系统的关系

| 复用 | 新增 | 移除 |
|------|------|------|
| 文档解析链路（WordParseState + OCR） | PageIndex 树索引构建 | 通用 ReAct 提示词 |
| FSM 生命周期（breathe 循环） | 审查清单生成器 | 意图分类器（predict:*） |
| 提示词系统（execute_prompt） | 逐项合规检查循环 | l2p PDDL 规划模块 |
| 工具注册体系（ToolRegistry） | 人工复核交互 | 领域模型记忆系统（替换为 PageIndex） |
| | 审查报告模板 | |

### 6.2 状态机映射

| State | 所属层 | 生命周期 | 类型 |
|-------|--------|---------|------|
| `BuildKnowledgeState` | 知识层 | Enter: 接收标准路径 → Execute: 解析+建树+落盘 → Exit: 更新版本 | 新增 |
| `GenChecklistState` | 清单层 | Enter: 接收方案类型 → Execute: LLM筛选条文 → Exit: 清单持久化 | 新增 |
| `ReviewPlanState` | 执行层 | Enter: 加载方案+清单 → Execute: 逐项检查循环 → Exit: 汇总结果 | 新增 |
| `HumanReviewState` | 复核层 | Enter: 展示存疑项 → Execute: 等待人工 → Exit: 记录复核 | 新增 |
| `GenReportState` | 报告层 | Enter: 加载结果 → Execute: 生成报告 → Exit: 标记完成 | 新增 |
| `WordParseState` | 工具 | 已有，不做改动 | 复用 |
| `ImgOCRState` | 工具 | 已有，不做改动 | 复用 |

    # 4. 记录判定结果（每条立即落盘，支持断点续传）

```
BuildKnowledge → GenChecklist → WordParse(方案) → ReviewPlan → HumanReview → GenReport
```

## 阶段七：架构验证

### 7.1 需求覆盖矩阵

| 需求 | 覆盖 | 实现方式 |
|------|------|---------|
| R1 标准知识库构建 | ✓ | BuildKnowledgeState + PageIndex 树索引 |
| R2 审查清单管理 | ✓ | GenChecklistState — LLM 按方案类型筛选条文 |
| R3 逐项合规检查 | ✓ | ReviewPlanState — 清单驱动 + PageIndex 导航 + LLM 判定 |
| R4 人工复核闭环 | ✓ | HumanReviewState — 存疑/不符合项 → 人工标注 → 更新判定 |
| R5 审查报告生成 | ✓ | GenReportState — 结构化报告 + 条文引用 + 归档 |
| NF1 可追溯 | ✓ | 每条判定 → clause_ref + plan_excerpt + reason |
| NF2 覆盖完整 | ✓ | 审查清单保证逐条遍历 |
| NF3 知识可更新 | ✓ | BuildKnowledgeState 可重复执行 |
| NF4 人工兜底 | ✓ | "存疑"判定 + HumanReviewState 强制人工确认 |

### 7.2 边界条件

| 条件 | 处理 |
|------|------|
| 标准文档无编号层级 | PageIndex 不依赖正则匹配，LLM 解析自然层级 |
| 方案内容覆盖不全 | 判定为"存疑"而非强行判定，进入人工复核 |
| 多标准冲突 | 审查清单标注来源，冲突标记"存疑" |
| 审查中断恢复 | 每条判定立即落盘，按 checklist_id 去重续传 |

### 7.3 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 清单项过多，审查耗时长 | 中 | 强条优先、异步执行+断点续传、后续支持批量判定 |
| LLM 判定准确率不足 | 高 | 置信度阈值自动升级"存疑"、不符合项强制人工确认、定期抽样评估 |
| PageIndex 生态成熟度 | 低 | 封装调用不直接依赖、保留降级方案（LLM 直接读条文全文） |

### 7.4 验证结论

9 项需求全覆盖，4 个边界条件有预案，3 个风险已识别并给出缓解措施。架构设计通过验证。
