# Codex / Claude Code / Kiro 配置说明

这份文档主要回答四件事：

1. 如何配置 `Codex`
2. 如何配置 `Claude Code`
3. 如何理解 VSCode 里的 `Codex` 插件配置
4. `Kiro` 有没有必要用

前提：你现在用的是**第三方中转站 API key**，不是官方直连 key。

---

## 1. 先说结论

如果你当前目标是：

```text
在本地开发里稳定使用大模型辅助编码
```

我建议你优先级这样排：

1. `Claude Code`：最适合当前这种“对代码库做理解、改动、解释、重构”的工作流
2. `VSCode Codex 插件`：适合补全、轻问答、编辑器内交互
3. `Kiro`：不是不能用，但更适合“规格驱动 / 项目范式化开发”，不一定适合你现在这种探索式、边看边改的工作流

一句话：

```text
日常代码理解和改造：Claude Code 优先
编辑器内补全和小交互：Codex 插件补充
规范化项目推进：Kiro 可选，不是必须
```

## 2. 配置第三方中转站时，你先要分清三件事

你现在不是用官方直连，所以要先分清：

### 2.1 API Key

例如：

```text
sk-xxxxx
```

这是身份凭证。

### 2.2 Base URL

第三方中转站通常不会走官方默认地址，而是会给你一个类似：

```text
https://your-proxy.example.com/v1
```

或者：

```text
https://api.xxx.com/openai/v1
```

这个 `base_url` 比 key 还重要，因为很多工具默认只知道官方地址。

### 2.3 Model Name

第三方中转站不一定用官方原始模型名，它可能要求你填：

```text
claude-opus-4-1
claude-sonnet-4
gpt-4.1
o4-mini
codex-mini-latest
```

也可能是平台自定义名字。

所以配置任何工具前，你都要先确认三件事：

```text
API_KEY 是什么
BASE_URL 是什么
MODEL 名字是什么
```

## 3. 如何配置 Claude Code

这里讲的是思路，不强依赖某一个壳子版本。

`Claude Code` 这类 CLI 工具，本质上需要：

- 模型供应商
- API key
- base url
- model 名称

如果你走第三方中转站，核心就是让它别走默认官方地址，而是走你的中转站。

### 3.1 最常见的配置方式：环境变量

很多 CLI 工具都支持环境变量配置。常见思路是这样的：

```powershell
$env:ANTHROPIC_API_KEY="你的第三方key"
$env:ANTHROPIC_BASE_URL="https://你的中转站地址"
```

如果这个工具兼容 OpenAI 风格接口，也可能是：

```powershell
$env:OPENAI_API_KEY="你的第三方key"
$env:OPENAI_BASE_URL="https://你的中转站地址"
```

然后再指定模型，例如工具自己的配置文件或启动参数里填：

```text
claude-sonnet-4
```

或者：

```text
claude-opus-4-1
```

这个名字必须以你中转站支持的为准。

### 3.2 如果工具支持配置文件

有些 CLI 会有自己的配置文件，比如：

```text
~/.config/...
```

或 Windows 下用户目录里的 app config。

那你本质上还是填这三样：

```text
provider/base_url/api_key/model
```

### 3.3 你真正要验证的不是“配置文件写了没”，而是三件事

1. 工具有没有成功读到 key
2. 请求有没有发到中转站的 base url
3. 模型名是不是中转站认可的名字

如果这三件里有一个不对，就会出现：

- 401：key 不对
- 404：base url 或 model 路径不对
- 400：模型名不支持
- 工具默认回退官方地址：说明 base url 没生效

### 3.4 我建议你的 Claude Code 配置原则

如果你是长期用第三方中转站，建议：

- 不要把 key 硬编码进项目仓库
- 优先用环境变量
- 单独记录一份“当前可用 base_url + model 名称”清单

例如你自己本地维护：

```text
Claude Code
base_url = https://xxx/v1
model = claude-sonnet-4
```

这样以后换壳子时，只改环境变量，不改项目代码。

## 4. 如何配置 Codex

这里分两层：

1. 你说的“Codex 工具/CLI”
2. VSCode 里的 `Codex` 插件

这两者不是一回事。

### 4.1 如果你说的是 Codex CLI / 命令行工具

那配置思路和上面一样，核心还是：

```text
API_KEY
BASE_URL
MODEL
```

如果它走 OpenAI 风格接口，通常就是：

```powershell
$env:OPENAI_API_KEY="你的第三方key"
$env:OPENAI_BASE_URL="https://你的中转站/v1"
```

然后模型可能填：

```text
gpt-4.1
o4-mini
codex-mini-latest
```

具体取决于你的中转站实际支持什么。

### 4.2 如果工具本身只认官方域名怎么办

这种情况下要看它是否允许：

- 自定义 `base_url`
- 自定义 provider
- 通过兼容 OpenAI API 的方式注入 endpoint

如果它完全写死官方接口，那第三方中转站就不一定能直接接进去。

所以你配置前先看这个工具是否支持：

```text
custom endpoint / api base / base url
```

### 4.3 如何验证 Codex 是否真的走了中转站

你要验证的不是“界面看起来能用”，而是：

1. 请求是否真的发到你的第三方地址
2. 返回的模型是否是你设定的那个
3. 是否出现官方登录绑定要求

如果工具强依赖官方账号体系，那和“纯 API key 中转站”不是一个路线。

## 5. VSCode 里的 Codex 插件怎么理解

你提到的是：

```text
VSCode 里的 Codex 插件
```

这里你要先分清：

```text
它到底是“OpenAI 官方账号型插件”
还是“支持自定义 API endpoint 的通用插件”
```

这两类差别很大。

### 5.1 如果它是官方账号型插件

那它通常更偏：

- 登录 OpenAI 账号
- 官方托管配置
- 不一定开放第三方 `base_url`

这种情况下，你的第三方中转站 key 不一定好接。

### 5.2 如果它支持自定义 API Endpoint

那你就按下面的思路配：

- API Key：填第三方 key
- Base URL / Endpoint：填第三方中转站地址
- Model：填中转站支持的模型名

在 VSCode 插件配置页里，最值得找的字段是：

```text
API Key
Base URL / Endpoint / API Base
Model
Provider
```

### 5.3 VSCode 插件里最常见的坑

#### 坑 1：插件支持模型名，但不支持自定义 base url

这样你填了 key 也没用，它还是请求官方地址。

#### 坑 2：插件支持自定义 base url，但内部某些功能仍走官方账号

比如聊天能走自定义，补全不能走自定义。

#### 坑 3：第三方中转站模型名和插件预置模型名对不上

例如插件只预置：

```text
gpt-4o
gpt-4.1
```

而你的中转站暴露的是：

```text
openai/gpt-4.1
```

这时就要看插件是否允许手填模型名。

### 5.4 我对 VSCode Codex 插件的建议

如果你主要想要的是：

```text
编辑器内补全 + 轻问答
```

那它可以当辅助工具。

但如果你想做的是：

- 跨文件理解代码库
- 解释复杂状态机逻辑
- 让模型帮你拆架构、写文档、重构函数

那单靠 VSCode 插件通常不如 `Claude Code` 这类更强的 coding agent 工具舒服。

也就是说：

```text
VSCode Codex 插件更像“编辑器能力增强”
Claude Code 更像“外部工程协作助手”
```

## 6. 第三方中转站下，最实用的配置策略

你现在既然走的是第三方中转站，我建议你把工具层分成两类：

### 6.1 命令行类工具

比如：

- Claude Code
- 各类 agent CLI

统一用：

```text
环境变量 + base_url + model
```

### 6.2 编辑器类工具

比如：

- VSCode 插件

优先选择那些明确支持：

```text
自定义 endpoint
手填 model
仅靠 API key 使用
```

不要优先选那种强依赖官方账号登录体系、但 API 配置很封闭的插件。

## 7. Kiro 有必要用吗

我的判断和你接近：

```text
Kiro 不是没用，但它更适合规格化、范式化、阶段目标清晰的开发。
```

你说的这些关键词很准确：

- PRD
- MVP
- 技术栈链路
- 明确任务边界
- 明确产出结构

这些都很适合 Kiro 这类工具。

### 7.1 Kiro 更适合什么场景

例如：

- 从 0 到 1 搭一个完整小应用
- 有比较清楚的产品目标和技术路线
- 希望按阶段推进：需求 -> 设计 -> 任务拆解 -> 代码实现
- 希望模型帮你维持“规格一致性”

这时候 Kiro 的体验通常会比较顺。

### 7.2 Kiro 不一定最适合你当前这种场景

你现在很多工作更像：

- 理解已有项目
- 读旧代码
- 对着真实 Word / OCR / 状态机逻辑排问题
- 边看边改边解释

这类工作不是强规格驱动，而是：

```text
探索式、诊断式、增量式开发
```

这种情况下，Kiro 可能会让你觉得：

```text
框架很完整，但有点重
```

因为它希望你先把：

- 目标
- 边界
- 计划
- 交付物

说清楚，再往下走。

但你现在经常是：

```text
先看代码 -> 发现问题 -> 追逻辑 -> 修局部 -> 再扩展理解
```

这更适合灵活一点的 coding assistant，而不是偏流程化的开发助手。

### 7.3 我的实际建议

如果你是现在这种工作流，我建议：

#### 日常主力

- `Claude Code`
- 或者你当前这种能直接读代码、解释逻辑、做修改的 agent 工具

#### 编辑器补充

- VSCode `Codex` / 其他补全插件

#### Kiro 的使用时机

只有在下面这些场景再上：

- 你准备单独起一个新模块
- 要写成比较完整的功能方案
- 要按照阶段推进一个较成型的功能
- 你希望工具帮你保持需求、设计、任务的一致性

所以我的建议不是：

```text
永远不用 Kiro
```

而是：

```text
现在没必要把 Kiro 当主力；等你做更规格化的项目推进时再上，会更舒服。
```
