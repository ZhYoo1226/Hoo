基于上述对话历史，分析任务信息需求，判断当前任务是否存在影响任务推进的**关键信息缺失或模糊点**。优先尝试自行推理或假设（不提问），只有在确实无法推进时才向用户提问。

## 判断规则
1. **需要提问 (meta:ask)**
   - 用户意图明确，但缺少必要参数（如时间、地点、对象、具体数值等）
   - 用户表述存在歧义，且无法从上下文消解
   - 后续行动存在多个分支，且无法通过默认值或常识决定
   - 提问应聚焦于**最少、最关键**的缺失信息，避免连串提问

2. **无需提问，继续执行 (predict:continue)**
   - 信息已足够制定行动方案
   - 缺失的信息可通过查询内部数据、外部知识或合理默认值补全
   - 用户刚刚拒绝了提问，或已明确表示“按你的判断做”

## 输出格式
**必须以 `### 分析任务信息需求` 开头**，然后输出一个 JSON 对象，并用 ```action 代码块包裹。

### 当需要提问时：
```action
{
    "action": "meta:ask",
    "question": "具体、可回答的提问内容（只问最必要的1-3个问题）"
}
```

### 当无需提问时：
```action
{
    "action": "predict:continue"
}
```

## 示例

**示例 1（需要提问）**
对话历史：用户说“帮我查一下明天的天气”
分析：缺少具体地点。无法确定默认城市（如未设置过）。→ 提问
输出：
### 分析任务信息需求
```action
{"action": "meta:ask", "question": "请问你想查询哪个城市的天气？"}
```

**示例 2（无需提问）**
对话历史：用户说“总结一下这个文档”，并已上传了文档内容。
分析：文档内容已提供，可直接总结。→ continue
输出：
### 分析任务信息需求
```action
{"action": "predict:continue"}
```

**示例 3（无需提问，用默认值）**
对话历史：用户说“订明天早上到上海的高铁”，历史中曾提到“常用出发地是杭州”。
分析：出发地可以从历史推断，用默认值即可。→ predict:continue
输出：
### 分析任务信息需求
```action
{"action": "predict:continue"}
```
