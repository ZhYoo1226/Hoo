基于上述对话历史，分析当前用户的回答是否包含对**某个未完成任务**的关键信息补充，且该信息未被处理。

# 判定逻辑
- 如果用户回答的内容**明确或隐含地**为某个任务提供了之前缺失的必要信息（如时间、地点、对象、数量、条件等），则认定为“信息补充”。
- 如果用户回答与当前任何待完成任务的信息需求无关，或仅为闲聊、确认、拒绝等，则认定为“无需处理”。

> 注意：任务应来自之前的对话上下文，你需要从历史中推断出 `task_id`（如有）。

# 输出格式
**必须以 `### 用户补充信息` 开头**，然后输出一个由 ` ```action ` 包裹的 JSON 对象。

## 情况1：用户正在补充任务信息
```action
{
    "action": "meta:clarify",
    "task_id": "12345678",
    "information": "提炼后的关键信息摘要（去除冗余，保留结构化要点）"
}
```

## 情况2：用户回复与任务信息补充无关
```action
{
    "action": "predict:continue"
}
```

- 对话历史：  
  用户: 帮我订张从杭州到上海的机票

### 用户补充信息
```action
{
    "action": "meta:clarify",
    "task_id": "20260518010101",
    "information": "出发日期：下周二"
}
```

- 对话历史：

